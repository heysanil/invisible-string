# Workflows & versions

Publishing a workflow compiles its four-pillar definition into a real eve agent
project, runs `eve build`, and uploads the resulting tarball to object storage.
The artifact is cached by a **content hash** — a digest of the pillar
configuration, the compiler version, the pinned eve version, and the build
environment — so an identical definition never rebuilds.

Each published version is immutable and gets its own compiled artifact. Because
the content hash fully determines the build, the same inputs always resolve to
the same cached tarball, and every version is addressable independently on the
worker pool.

## Version pinning

Versions matter because runs are bound to them. A session pins the exact
compiled agent version it started with, so editing and re-publishing a workflow
never changes the behaviour of runs already in flight — new versions apply to
new sessions.
