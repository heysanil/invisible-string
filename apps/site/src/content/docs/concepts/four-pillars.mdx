# The four pillars

A workflow in invisible-string is fully described by four decisions. Each is a
focused editor in the builder, and together they compile into a single runnable
agent — nothing more is needed, and nothing is hidden.

## Trigger

How the run starts. A trigger is one of `manual` (fired from chat), `webhook`,
`form`, or `slack`. Webhook and form triggers ingress through a per-workflow
token; Slack triggers route bound teams and channels through the connected
platform Slack app.

## Context

Everything the agent can reach at runtime: MCP connections (workspace or
personal) and authored skills with their attachments. Context is packaged
straight into the compiled agent, so a published workflow carries its tools with
it.

## Agent

The model behind the run. You pick a preset — `powerful`, `balanced`, or
`quick` — resolved against the workspace's provider and model allowlist, so a
workflow can never reference a model the workspace hasn't sanctioned.

## Instructions

The natural-language spec the agent follows. The instructions editor supports
`@`-references into trigger data (`@trigger.<path>`) and context — connections
and skills — with autocomplete and dry-run diagnostics that flag dangling
references before you publish.
